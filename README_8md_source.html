13 ## Overview
14 
15 This is an implementation of the core cryptographic routines
16 for the [Prio system](https://crypto.stanford.edu/prio/)
17 for the private computation of aggregate statistics:
18 > "Prio: Private, Robust, and Scalable Computation of Aggregate Statistics"<br>
19 > by Henry Corrigan-Gibbs and Dan Boneh<br>
20 > USENIX Symposium on Networked Systems Design and Implementation<br>
21 > March 2017
22 >
23 > Available online at:
24 > https://crypto.stanford.edu/prio/
25 
26 **Usage scenario.**
27 The library implements the cryptographic routines necessary
28 for the following application scenario:
29 Each client holds a vector of boolean values.
30 Each client uses the library to encode her private vector into two
31 encoded packets&mdash;one for server A and one for server B.
32 
33 After receiving shares from a client, the servers can use the routines
34 implemented here to check whether the client-provided packets are
35 well formed.
36 (Without this check, a single malicious client can corrupt the
37 output of the computation.)
38 
39 After collecting data packets from many clients, the servers
40 can combine their state to learn how many clients had the
41 *i*th bit of their data vector set to `true` and how many
42 clients had the *i*th bit of their data vector set to `false`.
43 As long as at least one of the two servers is honest
44 (i.e., runs the correct code),
45 the servers learn *nothing else* about the clients' data,
46 under standard cryptographic assumptions.
47 
48 For example, the *i*th bit of the data vector could indicate
49 whether the client ever visited the *i*th-ranked website
50 in the Alexa Top 500.
51 The servers would learn how many clients visited each of the
52 Top 500 websites *without learning* which clients visited
53 which websites.
54 
55 **Efficiency considerations.**
56 The code makes no use of public-key crypto, so it should
57 be relatively fast.
58 When each a data packet is of length *N*,
59 all arithmetic is modulo a prime *p* (we use an 87-bit prime by default),
60 and "elements" are integers modulo *p*,
61 the dominant costs of the system are:
62 * **Client compute:** O(*N* log *N*) multiplications
63 * **Client-to-server communication:** 2*N* + O(1) elements<br>
64 * **Server compute:** O(*N* log *N*) multiplications to check each packet<br>
65  (NOTE: Using an optimization we haven't yet implemented, we can
66  drop this cost to O(*N*) multiplications per packet.)
67 * **Server-to-server communication:** O(1) elements
68 * **Server storage:** O(*N*) elements

70 ## Running the code
71 
72 You must first install
73 [NSS/NSPR](https://developer.mozilla.org/en-US/docs/Mozilla/Projects/NSS)
74 with NSPR version 4.13.1 (or newer?) and NSS version 3.35 (or newer?),
75 [SCons](http://scons.org/) version 3.0.1 (or newer?), and
76 [msgpack-c](https://github.com/msgpack/msgpack-c) version 2.1.5 (or newer?).
77 
78 On Ubuntu Bionic (18.04LTS), you can install NSS and scons with:
79 
80  $ sudo apt install scons libnspr4-dev libnss3-dev
81 
82 and you will have to download [msgpack-c 2.1.5 or newer here](https://github.com/msgpack/msgpack-c/releases),
83 since the Ubuntu packages for msgpack are far out of date.

124 ## Optimizations and features not yet implemented
125 * **Server compute.**
126  By using a fast polynomial interpolation-and-evaluation
127  routine, we can reduce the cost of checking a single client
128  request from O(*N* log *N*) multiplications down to O(*N*)
129  multiplications, for a data packet of *N* items.
130 * **Differential privacy.**
131  It would be very straightforward to add some small amount of
132  noise to the final statistics to provide differential privacy.
133  If this would be useful, I can add it.
134 * **Misc.**
135  There are TODO notes scattered throughout code indicating
136  places for potential performance optimizations.
